# 代码风格规定

## 0. 序言

本规范旨在构建**高可读、易维护、强扩展**的 Godot C# 代码库。所有开发者应遵循以下黄金原则：

### 核心黄金原则

1.  **约定优于配置 (Convention over Configuration)**
    *   **默认即标准**：遵循项目既定的命名、目录结构和代码格式（如 Allman 风格）。除非有充分理由，否则不要引入额外的配置文件或特殊的命名变体。
    *   **减少决策成本**：让框架和规范的默认行为处理 90% 的场景，开发者只需关注那 10% 的特殊业务逻辑。

2.  **高内聚，低耦合 (High Cohesion, Low Coupling)**
    *   **单一职责**：每个类/方法只做一件事。利用“功能分组注释块”将相关逻辑物理聚合。
    *   **明确边界**：模块间通过接口或信号交互，严禁跨模块直接访问私有成员或硬编码节点路径。

3.  **保持简洁 (KISS)**
    *   **清晰优于聪明**：拒绝晦涩的语法糖和过度设计。代码应像散文一样直白，让新成员能瞬间理解。
    *   **消除噪音**：移除无用代码和冗余注释，让核心逻辑凸显。

4.  **创造价值 (Create Value)**
    *   **结果导向**：代码必须服务于游戏玩法、性能或稳定性。避免为了炫技而编写的“死代码”。
    *   **可维护性即资产**：今天的整洁代码是明天快速迭代的基础。

5.  **拒绝重复 (DRY - Don't Repeat Yourself)**
    *   **唯一真理源**：重复的逻辑必须抽象为通用方法或基类。任何修改都应在唯一源头进行，杜绝复制粘贴带来的维护灾难。

6.  **拥抱变化 (Embrace Change)**
    *   **为扩展而生**：沙盒游戏需求多变，代码结构需预留扩展点，确保新功能接入时不破坏现有系统。

## 1. 功能分组注释块

类内为相同功能服务的属性与方法，使用以下格式分隔：

```
// ================================================================================
//                                  某某方法
// ================================================================================
```

## 2. 注释规范

### 类、方法、属性
尽可能为每个类、方法、属性添加注释。

### 函数内代码

**重要逻辑与局部变量**（包括for循环中的i）：注释放在前一行
```
// 注释
重要逻辑或局部变量
```

**其他情况**：注释写在语句后面
```
函数() // 注释
```

## 3. 命名规范

鼓励使用 `get`/`set` 等谓语词以助于理解。

## 4. 输出规范（Godot引擎）

本项目为2D沙盒游戏，采用Godot引擎开发。**禁止使用** `Console.WriteLine`，该方法不会在Godot控制台中显示。

应使用以下Godot API进行输出：

- `GD.Print()` - 普通信息输出
- `GD.PushError()` - 错误信息输出（红色显示）
- `GD.PushWarning()` - 警告信息输出（黄色显示）

## 5. 空行规范

用空行明晰结构与逻辑，提高代码可读性。

### 相近功能(私有字段与公开属性)
- 相近功能与其对应的相近功能之间用**一个空行**隔开
- 相近功能作为一个整体，与其他代码用**两个空行**隔开
- 相近功能如私有字段与公开属性、方法重载等，也应遵循此规范。

**示例**：
```csharp

// 其他代码...


/// <summary>
/// 实例的唯一标识符
/// </summary>
private int _id;

/// <summary>
/// 获取实例的唯一标识符
/// </summary>
public int Id => _id;


/// <summary>
/// 所属世界实例
/// </summary>
private World _ownerWorld;

/// <summary>
/// 获取所属世界实例
/// </summary>
public World OwnerWorld => _ownerWorld;


// 其他代码...
```

## 5. 花括号规范

在本项目中，所有控制语句（如 if、for、while、switch 等）和方法、类的花括号必须采用 Allman 风格，即左花括号 { 单独成行，并与对应的关键字对齐。

**✅正确示例**：
```csharp
if (condition)
{
    // 代码
}

// 其他代码...
```

此风格符合 Microsoft 官方 C# 编码约定，有助于提升代码可读性、维护性及团队协作一致性。禁止将左花括号与条件或声明写在同一行。格式应通过 .editorconfig 和 IDE 自动化工具统一管理。

**❌错误示例**：
```csharp
if (condition){//应当放在新行❌❌❌
    // 代码
}

// 其他代码...
```
