计数器系统文档

线程安全的自增计数与持久化解决方案

系统概述

计数器系统提供线程安全的自增计数功能，支持数据持久化，适用于游戏开发中实体ID分配、资源计数等场景。

Counter 计数器类

提供线程安全的自增计数功能，用于生成唯一标识符。

CounterHistoryManager 历史数据管理器

管理计数器的持久化数据，支持存档保存与恢复。

创建管理器

创建 CounterHistoryManager 实例，指定文件路径用于持久化存储。管理器会自动从文件加载历史数据到「历史数据表」，若文件不存在则创建空表。每个存档应使用独立的管理器实例。

创建计数器

创建 Counter 实例，指定计数器名称（持久化标识）和可选的管理器实例。计数器会自动生成 UID（运行时标识），从管理器获取历史数据初始化，并向管理器注册。

使用计数器

调用 GetAndIncrement() 方法获取唯一标识符。每次调用返回当前值并按步长自增。推荐使用此方法而非直接读取 CountValue。计数器操作是线程安全的。

保存数据

调用 manager.Save() 方法将所有已注册计数器的当前值保存到文件。数据以 JSON 格式存储，键为计数器名称，值为当前计数值。应在游戏存档或退出时调用。

释放资源

调用 counter.Dispose() 方法释放资源。计数器会从管理器的注册表中注销，并从全局实例映射表中移除。不再使用的计数器应及时释放以避免内存泄漏。

深入流程

第一阶段：初始化

▼

创建管理器实例

指定文件路径（如 "save1/counters.json"），自动从文件加载历史数据到「历史数据表」，若文件不存在则创建空表。

创建计数器实例

  • 生成唯一 UID（运行时标识，每次创建都不同）
  • 指定计数器名称（持久化标识，恒定不变）
  • 从管理器查询历史数据：有历史数据则采用，无历史数据则使用 long.MinValue + 1 初始化
  • 自动向管理器注册（加入「注册表」）
  • 加入全局实例映射表（支持 UID 查询）

第二阶段：运行使用

▼

在游戏运行过程中使用计数器生成唯一标识符。

方法 说明
GetAndIncrement() 返回当前值并按步长自增。推荐使用此方法获取唯一标识符。
GetAndIncrementOne() 返回当前值并自增 1。
Increment() 仅自增，不返回值。
CountValue 当前计数值（下一个待使用的序号）。不推荐直接读取，应使用 GetAndIncrement()。
GetCounterByUID(uid) 静态方法，通过 UID 查询计数器实例。

第三阶段：持久化

▼
  • 调用 manager.Save()
  • 遍历注册表中所有计数器
  • 获取每个计数器的当前值
  • 序列化为 JSON 写入文件

第四阶段：资源释放

▼
  • 调用 counter.Dispose()
  • 从管理器注销（移出注册表）
  • 从实例映射表移除

设计要点

双标识符设计

名称（CounterName）：恒定不变，用于持久化，跨存档保持一致

UID：运行时生成，用于实例查询，每次启动都不同

long.MinValue 的特殊语义

  • 不作为有效序号
  • GetHistoryValue() 返回此值表示「无历史数据」
  • 计数器初始化时检测到此值则使用 long.MinValue + 1

线程安全

所有读写操作都使用锁保护

存档隔离方案

每个存档使用独立的管理器实例，不同存档指向不同文件路径，计数器数据互不干扰

注意事项

⚠️ long.MinValue

long.MinValue 不作为有效序号，仅表示「无历史数据」或错误状态。计数器初始化值为 long.MinValue + 1。

⚠️ Manager = null

管理器参数可为 null，此时计数器不进行持久化，适用于临时计数场景。

⚠️ CountValue

CountValue 中的值是"下一个待使用"的序号，直接读取通常无意义。

⚠️ SetValue()

SetValue() 会破坏自增语义，可能导致 ID 重复，仅在特殊场景使用。

⚠️ Save()

记得在适当的时机调用 Save() 保存数据，如游戏退出或存档时。

⚠️ Dispose()

计数器不再使用时应调用 Dispose() 释放资源。
