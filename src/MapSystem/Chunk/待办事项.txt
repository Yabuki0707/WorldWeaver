## 设计模式简洁叙述
### 状态模式 每个状态一个类，类里封装该状态的行为和转换规则
- 状态机只负责"当前是哪个状态"
- 具体做什么、接下来去哪，由各状态类自己决定
- 新增/修改状态 → 增/改一个类，不影响其他类
### 策略模式 把"怎么做"抽象成接口，具体实现可替换
- 把回调逻辑从状态机中抽离出来
- 同一个接口可以有不同实现（如同步/异步策略）
- 运行时可切换策略，不改状态机代码
### 在这个场景的应用
```
原设计：状态 + 转换规则 + 回调 → 全硬编码在一个静态数组
里

新设计：
┌─────────────┐     ┌─────────────┐
│ 状态机引擎   │────▶│ StateHandler│ (状态模式)
│ (路由导航)   │     │ (每个状态一个类)│
└─────────────┘     └─────────────┘
                           │
                           ▼
                    ┌─────────────┐
                    │ICallbackPolicy│ (策略模式)
                    │ (回调执行策略) │
                    └─────────────┘
```
核心思路 ：把"谁来做"和"怎么做"分离，状态机只管流转，具体业务逻辑下沉到各 Handler 类。
