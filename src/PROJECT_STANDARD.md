# 项目标准规范

**版本：1.3**  
**出版时间：2026年3月29日**

## 0. 序言

本规范旨在构建**高可读、易维护、强扩展**的 Godot C# 代码库。所有开发者应遵循以下黄金原则：

### 核心黄金原则

1. **约定优于配置 (Convention over Configuration)**
    * **默认即标准**：遵循项目既定的命名、目录结构和代码格式（如 Allman 风格）。除非有充分理由，否则不要引入额外的配置文件或特殊的命名变体。
    * **减少决策成本**：让框架和规范的默认行为处理 90% 的场景，开发者只需关注那 10% 的特殊业务逻辑。

2. **高内聚，低耦合 (High Cohesion, Low Coupling)**
    * **单一职责**：每个类/方法只做一件事。利用“功能分组注释块”将相关逻辑物理聚合。
    * **明确边界**：模块间通过接口或信号交互，严禁跨模块直接访问私有成员或硬编码节点路径。

3. **保持简洁 (KISS)**
    * **清晰优于聪明**：拒绝晦涩的语法糖和过度设计。代码应像散文一样直白，让新成员能瞬间理解。
    * **消除噪音**：移除无用代码和冗余注释，让核心逻辑凸显。

4. **创造价值 (Create Value)**
    * **结果导向**：代码必须服务于游戏玩法、性能或稳定性。避免为了炫技而编写的“死代码”。
    * **可维护性即资产**：今天的整洁代码是明天快速迭代的基础。

5. **拒绝重复 (DRY - Don't Repeat Yourself)**
    * **唯一真理源**：重复的逻辑必须抽象为通用方法或基类。任何修改都应在唯一源头进行，杜绝复制粘贴带来的维护灾难。

6. **拥抱变化 (Embrace Change)**
    * **为扩展而生**：沙盒游戏需求多变，代码结构需预留扩展点，确保新功能接入时不破坏现有系统。

## 1. Godot 场景树节点的最佳实践

### 1.0 模型

🔄Node 生命周期：

`构造函数` → `_EnterTree()` → `_Ready()` → `_ExitTree()` →（可再次进入）

### 1.1 一句话原则

**Node 通常应该：进 → 跑 → 出 → 再进 → 再跑（无副作用）**

> 原因：Node 在引擎中是可复用运行体，而非一次性对象

### 1.2 职责划分

#### 构造函数

**应该只做一件事：**
- 字段默认值

**避免：**
- ❌ 访问节点
- ❌ 注册系统
- ❌ 任何运行时逻辑

> 原因：此时尚未进入场景树

#### _EnterTree()

**调用特性：** ✔ 每次进入都会调用

**用于：**
- 接入外部系统（注册、监听等）

**特点：**
- 父节点存在
- 子节点不保证可用

> 原因：该阶段是运行环境入口，必须支持重复执行

#### _Ready()

**调用特性：** ⚠️ 默认只调用一次

**用于：**
- 获取子节点
- 绑定信号
- 初始化内部结构

> 原因：依赖完整节点结构，且不会在重进时再次触发

#### _ExitTree()

**用于：**
- 撤销 `_EnterTree` / `_Ready` 的所有影响
- 注销系统、解绑信号、清理引用

**避免：**
- ❌ 破坏对象状态（必须可再次进入）

**补充：**
- 生命周期结束 = 离开场景树
- 所有运行时逻辑必须在此清理

> 原因：Node 没有销毁回调，只有离开场景

### 1.3 最佳规则

#### Enter / Exit 需对称

**Enter 做的事，Exit 必须全部撤销**

所有影响“再次进入”的逻辑，应该写在 `_EnterTree()` / `_ExitTree()`内，并需成对出现。

> 原因：否则会产生状态残留或重复注册

#### Ready 不是重复入口

**❌ 不要写“每次进入都要执行”的逻辑**

> 原因：`_Ready` 不会在重进时调用

#### QueueFree = 对象已失效

```csharp
node.QueueFree();
// 之后禁止再使用 node
```

> 原因：对象已进入销毁流程，行为不再可靠

### 1.4 构造函数限制

- **必须无参**
- **不允许传依赖**

> 原因：Node 由引擎实例化

### 1.5 关于 Dispose

**Node 不走 Dispose**

- 不用它管生命周期
- 不保证被调用
- 不写逻辑

> 原因：生命周期由引擎控制

### 1.6 关键划分（Enter vs Ready）

| 方法             | 调用特性    | 用途             |
|----------------|---------|----------------|
| `_EnterTree()` | ✔ 会重复调用 | 放“每次进入都要执行的逻辑” |
| `_Ready()`     | ⚠️只调用一次 | 放“只需初始化一次的结构”  |

> 原因：调用次数不同，职责必须分离

### 1.7 总结

**生命周期不是“什么时候写代码”，而是代码能不能在第二次运行时不出问题**

- 场景树重复进入的保障，依赖于 `_EnterTree()` / `_ExitTree()` 的代码
- 写在 `_Ready()` 的代码，只保证初始化一次

**也就是：**

- 把 Node 移出再加回来，行为必须完全一致

> 参考:https://school.gdquest.com/glossary/function_init
## 2. 结构体使用规范

### 2.1 使用建议

如果一个类型的实例较小，通常生存期较短，或通常会嵌入在其他对象中，优先考虑定义为 `struct` 而不是 `class`。

**🤤一般而言，如果你并不清楚，请通通用类**

### 2.2 禁止滥用结构体

除非一个类型同时满足以下全部条件，否则不要定义为 `struct`：

- 它在逻辑上表示单个值，语义接近 `int`、`double` 这类基元值类型。
- 它的实例大小小于 16 字节。
- 它是不可变的。
- 它不会被频繁装箱。

### 2.3 设计要求

- `struct` 应优先设计为只读值对象，避免暴露可变状态。
- 如果类型需要身份语义、共享引用、可空语义扩展、大量状态或后续继承扩展，应使用 `class`。
- 如果一个类型不满足上述全部条件，即使实例看起来不大，也不要为了“少一次分配”而强行改成 `struct`。

> 参考:https://learn.microsoft.com/zh-cn/dotnet/standard/design-guidelines/choosing-between-class-and-struct

## 3. `int x, int y` 与 `Vector2I` 的最佳实践

> Vector2I官方文档:https://docs.godotengine.org/zh-cn/4.x/classes/class_vector2i.html#

### 3.1 一句话原则

内部用 `int x, int y`，对外与引擎交互用 `Vector2I`，并提供重载桥接。

> 原因：`Vector2I` 表达坐标语义，`int x, int y` 具备微弱性能优势且不依赖于Godot引擎。

### 3.2 分类讨论

#### 内部计算

**推荐：**

- 💻在高频计算、循环、边界判断、下标换算等底层逻辑中，优先使用 `int x, int y`。
- 🍬在保持唯一实现的前提下，酌情补充 `Vector2I` 重载作为语法糖入口。

```csharp
private Chunk GetChunk(int x, int y)
{
    ...
}

public Chunk GetChunk(Vector2I pos)
{
    return GetChunk(pos.X, pos.Y);
}
```

**避免：**

- 🧶在热路径中把 `Vector2I` 当作纯计算中间变量反复创建、传递、拆解。

> 原因：底层计算关注数值处理本身。

#### Godot 引擎交互或抽象层级较高的场景

**推荐：**

- 直接使用 `Vector2I`。

**适用于：**

- `TileMap` / `Grid` 等 Godot 坐标接口。
- 字典键、坐标集合、系统间传参。
- 明确表达“坐标”的接口与抽象层代码。

**避免：**

- 为统一底层写法而强行改为 `int x, int y`。

> 原因：`Vector2I` 更符合引擎与接口语义。

### 3.3 过渡原则

**推荐：**

- 上层使用 `Vector2I`。
- 下层使用 `int x, int y`。
- 在边界处完成一次转换。

**避免：**

- `Vector2I` 与 `int` 在同一层反复来回转换。
- 上下层表达混乱。

> 原因：减少噪音与不必要开销。

### 3.4 取舍原则

推荐优先使用 `Vector2I` 保持语义清晰，仅在明确属于热路径或底层计算时使用 `int x, int y`。

> 原因：可读性与一致性通常比微弱性能更重要。

## 4. 输出规范（Godot 引擎）

本项目为 2D 沙盒游戏，采用 Godot 引擎开发。

**禁止使用** `Console.WriteLine`，该方法不会在 Godot 控制台中显示。

### 4.1 输出 API 选择

应使用以下 Godot API 进行输出：

#### GD.Print()
🙂普通或调试信息输出。

#### GD.PushError()
☹️错误信息输出。

#### GD.PushWarning()
😨警告信息输出。

### 4.2 错误输出说明

Godot 的报错本身会附带调用栈，因此错误输出的重点不是写栈里已经有的信息，而是补充**足够直观的上下文信息**。

错误语句应当满足以下要求：

- 必须包含**系统名**或**模块名**。
- 必须包含发生错误的**实例信息**。
- 必须包含**关键数据**或**关键参数**。
- 必须让人一眼看懂“谁出了什么错，错在什么数据上”。

栈与报错内容的区别:

- 栈是**技术信息**
- 日志是**业务信息**

### 4.3 推荐格式

推荐参考以下格式：

```csharp
$"[系统名或模块名]:某实例发生了错误，数据为{data}的某某错误。"
```

例如：

```csharp
GD.PushError($"[ChunkSystem/ChunkData]:Chunk {uid} 在设置 Tile 时发生错误，输入的 localPosition={localPosition} 非法。");
GD.PushWarning($"[MapSystem/ChunkManager]:Chunk {chunkPosition} 加载异常，返回的数据长度为{tileCount}。");
```

### 4.4 禁止事项

- 不要只输出“出错了”“参数非法”这类无上下文信息的语句。
- 不要省略实例标识、模块名或关键数据。
- 不要为了简短而牺牲可读性。

> 原因：Godot 已提供堆栈，业务层输出应专注于补齐语义信息与定位信息。

> 参考:https://hamidmosalla.com/2024/10/14/csharp-logging-best-practices-to-improve-troubleshooting

## 5. 驱动模型

### 5.1 一句话原则

系统只有两种基础驱动方式：主动检查与变化通知。判断标准不在于是否使用状态机、调度器或事件总线，而在于谁决定“什么时候触发逻辑”。

> 原因：驱动方式决定系统的节奏控制、模块耦合与调试成本。

### 5.2 主动检查（Polling / 主循环驱动）

#### ⚙️本质

系统按固定节奏主动检查状态，并在满足条件时执行逻辑。

```csharp

public override void _Process(double delta)
{
    Update();
}

public void Update()
{
    if (condition) DoSomething();
    // ...
}

```

#### 🎯适用场景

- 游戏主循环（`_Process` / Tick）
- 定时扫描
- AI 行为评估
- 冷却、超时、清理等周期性逻辑

#### ✨特点

- ✔ 执行顺序稳定
- ✔ 节奏完全可控
- ✔ 易于调试与限流

#### 🤔代价

- ❌ 可能存在过多的无效检查
- ❌ 检查范围过大时会浪费性能

### 5.3 变化通知（Event / Signal 驱动）

#### ⚙️本质

当状态发生变化时，主动发出通知，由订阅方执行响应逻辑。

```csharp
OnHealthChanged += HandleDamage;
```

#### 🎯适用场景

- 事件系统（`event` / `delegate` / `signal`）
- 观察者模式
- UI 刷新（数据变化后更新界面）
- 跨模块状态同步

#### ✨特点

- ✔ 只在变化时触发，避免无效检查
- ✔ 模块之间更容易解耦
- ✔ 响应通常更及时

#### 🤔代价

- ❌ 执行顺序不如主循环直观
- ❌ 调用链更难追踪
- ❌ 滥用后容易形成事件链耦合

> 原因：是否执行由状态变化决定，而不是由外层循环统一调度。

### 5.4 状态机、调度器与任务队列不是驱动方式

状态机、调度器、任务队列等都不是驱动方式，它们只是组织逻辑的结构。

```csharp
while (queue.TryDequeue(out var task))
{
    task();
}
```

上例中：

- 外层 `while` 是主动检查
- 内层 `task()` 的内容可能是变化触发后的响应

结构解决“如何组织逻辑”，驱动方式解决“何时触发逻辑”。

### 5.5 工程原则

**推荐：**

- 主动检查负责控制节奏
- 变化通知负责传播变化
- 在主循环中推进系统状态，在状态变化时对外发出通知

推荐结构：

```text
Update（主动检查）
→ 状态发生变化
→ 发出通知（事件 / Signal）
→ 其他系统响应
```

> 原因：这样既能保持节奏可控，又能避免模块间硬耦合。

### 5.6 避免极端

- 🔩所有逻辑都用主动检查，导致每帧扫描全部状态
- 🕸️所有逻辑都用变化通知，导致执行顺序不可控
- 🧪用事件链替代正常的流程控制
- 🔀用轮询去代替本该即时响应的状态变化传播

> 原因：单一驱动方式覆盖全部场景，通常只会放大它的缺点。
